# Trigger Zone Capture System - 開発コンセプト概要

## 📋 プロジェクト目標

Unityでアバターの歩行アニメーションを特定区間で連続撮影し、その画像シーケンスを別プロジェクトのMediaPipeで処理するシステムの構築。

## 🎯 核心となるアイデア

**「トリガーゾーン方式」による撮影制御**

従来の複雑な同期コードを避け、空間上の透明な壁（トリガーゾーン）を利用してアニメーション実行と撮影タイミングを制御する手法。

### システムフロー
```
[UIスタートボタン押下]
   ↓
[キャラクター配置(0,0,-5) + 歩行アニメーション開始]
   ↓
[Z軸方向に移動開始]
   ↓
[(0,0,-3) トリガー通過] → 📷 撮影開始
   ↓
[移動継続 + 撮影継続]
   ↓
[(0,0,3) トリガー通過] → 📷 撮影終了
   ↓
[移動継続]
   ↓
[(0,0,5) トリガー通過] → 🛑 全停止
```

## 💡 この方式のメリット

1. **シンプルな同期機構**: コードでフレーム単位の同期を取る必要がない
2. **視覚的な調整**: トリガーゾーンの位置を動かすだけで撮影区間を変更可能
3. **デバッグの容易性**: 壁を可視化すれば動作確認が簡単
4. **拡張性**: 複雑な経路や複数カメラへの対応が容易

## 🏗️ システム構成

### 4つの主要コンポーネント

#### 1. **WalkThroughCaptureManager** (マスターコントローラー)
- 全体統括・初期化処理
- UI連携
- 各コンポーネントの参照管理

#### 2. **CharacterMover** (キャラクター制御)
- Transform移動 (0,0,-5 → 0,0,5)
- Animatorによる歩行アニメーション再生
- 移動とアニメーションの同時実行

#### 3. **TriggerZone** (空間トリガー) × 3個
- **Zone 1**: (0,0,-3) - 撮影開始トリガー
- **Zone 2**: (0,0,3) - 撮影終了トリガー
- **Zone 3**: (0,0,5) - 全停止トリガー

#### 4. **FrameCapturer** (撮影システム)
- 連続スクリーンショット撮影
- 指定フレームレート（推奨30fps）での画像保存
- 連番ファイル管理

## 🔑 重要な技術ポイント

### アニメーションと移動の同時実行
```
❌ 誤解: Transform移動 = ただの平行移動（アニメーションなし）
✅ 正解: Transform移動 + Animation再生 = 歩きながら前進
```

- `Transform.position` で位置を移動（Z座標の変化）
- `Animator.Play("Walk")` で歩行モーション再生
- 両者を同時実行することで自然な歩行を実現

### 撮影の同期方式

**方式1: Time.captureFramerate（推奨）**
```csharp
Time.captureFramerate = 30;  // 固定フレームレート
```
- メリット: 処理落ちしても全フレーム確実に撮影
- デメリット: リアルタイム性喪失（スローモーション的に動く）

**方式2: リアルタイム撮影**
- メリット: 実時間で動作確認可能
- デメリット: 処理が重いとフレーム落ちの可能性

## 📦 MediaPipe連携の考慮点

### 出力仕様
- **フォーマット**: PNG（無圧縮・透過対応）またはJPG（軽量）
- **解像度**: 640×480 or 1280×720（MediaPipe入力要件に合わせる）
- **フレームレート**: 30fps推奨
- **ファイル命名**: 連番管理（例: `frame_0001.png`）

### 背景処理
- グリーンバック/単色背景 or 透過PNG出力
- MediaPipeの人物検出に適した背景設定

### カメラアングル
- ポーズ推定に適した角度（正面・側面など）
- アバター全身がフレーム内に収まる設定

## 🤔 設計上の検討事項

### A. キャラクター管理
- **案1**: Prefabから毎回生成（クリーン）
- **案2**: シーン配置済み（軽量、要リセット処理）

### B. アニメーション種別
- **その場歩き**: 足だけ動く → Transform移動と組み合わせる
- **前進歩き**: Root Motion使用 → より複雑な制御が必要

### C. 移動速度とアニメーション速度の同期
```
移動速度 = 2 m/s
アニメーション速度 = 1.0x
→ 足の滑りが発生しないか要検証
対策: animator.speedで調整
```

### D. トリガーゾーン配置
- **案1**: Manager側で自動生成
- **案2**: 手動でシーン配置（Inspector調整容易）

## 🎨 Public要素の設計

各クラスで公開すべきパラメータ：

### WalkThroughCaptureManager
- UI参照（Button、StatusText）
- キャラクター設定（Prefab、アニメーション名）
- 移動設定（開始/終了位置、速度）
- 撮影設定（カメラ、解像度、FPS、出力先）
- トリガーゾーン参照

### CharacterMover
- Animator参照
- 移動速度・方向
- 状態フラグ（isMoving、isAnimating）

### TriggerZone
- ゾーンタイプ（CaptureStart/CaptureEnd/Stop）
- Gizmo表示設定（デバッグ用）
- イベント通知機構

### FrameCapturer
- カメラ参照
- 解像度設定
- 出力フォルダパス
- 撮影状態（isCapturing、フレームカウント）

## 📝 次のステップ

実装前に決定すべき事項：

1. ✅ キャラクターの生成方法（Prefab or シーン配置）
2. ✅ アニメーションタイプ（その場歩き or 前進歩き）
3. ✅ 撮影モード（リアルタイム or 固定フレームレート）
4. ✅ トリガーゾーン配置方法（自動生成 or 手動配置）

## 🌟 この設計の革新性

従来の「フレーム単位での厳密な同期」という複雑なアプローチから、「空間ベースのイベント駆動」というシンプルで直感的なアプローチへの転換。

これにより、コードの可読性・保守性・拡張性が大幅に向上し、非プログラマーでも視覚的に撮影区間を調整できるシステムが実現可能になる。
